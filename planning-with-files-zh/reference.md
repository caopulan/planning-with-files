# 参考：Manus 上下文工程原则

本技能基于 Manus 的上下文工程原则。Manus 是一家 AI Agent 公司，于 2025 年 12 月被 Meta 以 20 亿美元收购。

## Manus 的 6 条原则

### 1. 文件系统作为外部记忆

> “Markdown 是我在磁盘上的‘工作记忆’。”

**问题：** 上下文窗口有上限。把所有内容塞进上下文会降低性能并增加成本。

**解决方案：** 把文件系统当作无限记忆：
- 把大内容存进文件
- 上下文里只保留路径
- 需要时再“查阅”信息
- 压缩必须可逆

### 2. 通过重复操控注意力

**问题：** 大约 50 次工具调用后，模型会忘记最初目标（“中间遗忘”效应）。

**解决方案：** 保持一个会被反复读取的 `task_plan.md` 文件：
```
上下文开头：[原始目标 - 很远，已被遗忘]
...很多工具调用...
上下文末尾：[最近读过的 task_plan.md - 获得注意力！]
```

在每次决策前读计划文件，让目标进入注意窗口。

### 3. 保留失败痕迹

> “错误恢复是展现真实自主行为的最清晰信号之一。”

**问题：** 本能会隐藏错误、静默重试。这会浪费 token 并丢失学习。

**解决方案：** 把失败行为保留在计划文件里：
```markdown
## 遇到的错误
- [2025-01-03] FileNotFoundError: config.json not found → 已创建默认 config
- [2025-01-03] API timeout → 使用指数退避重试后成功
```

模型在看到失败记录时会更新内部理解。

### 4. 避免少样本过拟合

> “千篇一律会滋生脆弱性。”

**问题：** 重复的行动-观察对会导致漂移与幻觉。

**解决方案：** 引入可控的变化：
- 稍微变化措辞
- 不要盲目复制粘贴模式
- 对重复任务进行再校准

### 5. 用稳定前缀做缓存优化

**问题：** 代理是输入密集型（100:1 比例）。每个 token 都有成本。

**解决方案：** 为缓存命中做结构化：
- 静态内容放在最前
- 只追加上下文（不要修改历史）
- 一致的序列化方式

### 6. 只追加的上下文

**问题：** 修改之前的消息会使 KV-cache 失效。

**解决方案：** 永远不要修改先前消息，只追加新信息。

## 代理循环

Manus 在一个持续循环中工作：

```
1. 分析 → 2. 思考 → 3. 选择工具 → 4. 执行 → 5. 观察 → 6. 迭代 → 7. 交付
```

### 循环中的文件操作：

| 操作 | 何时使用 |
|-----------|-------------|
| `write` | 新建文件或整体重写 |
| `append` | 逐步追加内容 |
| `edit` | 更新特定部分（复选框、状态） |
| `read` | 决策前复查 |

## 关键引述

> “对于复杂任务，我会把笔记、代码和发现写进文件，以便在工作中随时引用。”
